들어가며

“태블로를 바꿀 수 있을까요?”

한 고객사의 문의에서 시작됐다. 온프레미스 S/4HANA를 운영 중인데, Tableau를 Amazon Quick으로 대체하고 싶다고 했다. 조건이 붙어 있었다 — SAP 실시간 데이터여야 하고, AI가 장표를 자동으로 만들어줘야 한다는 것.

기술 검토를 하다 보면 이런 질문은 문서만으로 답하기 어렵다. 공식 문서는 “된다”고 쓰여 있지만, 온프레미스 SAP와 SaaS인 Quick 사이에는 네트워크, 인증서, 인증, 권한이라는 현실이 놓여 있다. 그래서 질문을 바꿔 던졌다 — 지금 내 손에 있는 것들로, 오늘 저녁 안에 직접 붙여볼 수 있을까?

재료는 세 가지였다. 온프레미스 S/4HANA 2023 데모 서버, Amazon Quick 데스크탑 앱, 그리고 Claude Code로 개발해 온 ABAP ADT 기반 MCP 서버. 이 문서는 그 세 가지가 처음 연결된 한 시간의 기록이다.

요약

세 문장으로 먼저

  • 연결. 온프레미스 S/4HANA와 Amazon Quick을 커스텀 MCP 서버(stdio)로 연결했다. 이관도, VPN도, 유효 인증서도 필요 없었다 — 자연어 요청만으로 Quick의 AI가 SAP 데이터를 실시간 조회했다.
  • 산출. FI 전표(ACDOCA) 집계로 재무 현황·손익계산서(P&L) 등 대시보드 3종이 한 세션 안에 만들어졌다. 요청은 전부 자연어였고, 코드는 한 줄도 직접 쓰지 않았다.
  • 통제와 검증. 민감 테이블 접근은 서버측 차단 목록이 막았고, 사용자 승인 후 개인정보 미포함 집계만 허용됐다. 그리고 AI가 만든 P&L에서 사람 검증이 집계 누락을 찾아냈다 — 자동화의 끝에는 여전히 검증이 있다.
01Amazon Quick

BI를 품은 AI 어시스턴트

먼저 이름부터 정리하자. Amazon Quick과 QuickSight는 어떤 관계일까. AWS 공식 문서의 표현을 그대로 옮기면 —

“Amazon Quick은 Amazon QuickSight에서 진화했다. QuickSight는 Quick 안의 한 기능, Amazon Quick Sight로 계속된다.”

원문: “Amazon Quick evolved from Amazon QuickSight. QuickSight continues as Amazon Quick Sight, a feature within Quick.” — What is Amazon Quick? ↗

즉 QuickSight(BI)가 사라진 게 아니라, Amazon Quick이라는 AI 어시스턴트 플랫폼의 한 구성요소로 편입됐다. Quick은 자연어 채팅으로 동작하며, 그 안에 다음이 들어 있다.

구성요소역할
Quick SightBI·대시보드 (구 Amazon QuickSight)
Quick FlowsAI 워크플로우 자동화
Quick Automate에이전트 기반 업무 프로세스 자동화
Quick Index사내 문서·데이터 연결 (응답 그라운딩)
Quick Research웹 + 사내 데이터 리서치 리포트
Apps in Quick자연어로 웹앱 생성

이번 실증에서 쓴 것은 Quick 데스크탑 앱의 채팅 + 커넥터(MCP) 조합이다. “대시보드를 만들어줘”라고 말하면, Quick의 AI가 연결된 데이터 소스에서 데이터를 조회하고 집계해 시각화 산출물을 만든다.

온프레미스는 안 되지 않나? — 흔한 오해다. AWS 문서는 SAP 연동의 전제조건을 “Quick과 SAP 시스템 간 네트워크 연결”로만 정의하며, SAP가 AWS 위에서 구동되어야 한다는 요건은 없다. 클라우드 이관 없이, 온프레미스를 유지한 채 연결만 하면 된다. (SAP workload integrations ↗)
02연결

표준 커넥터의 벽, 그리고 MCP

처음엔 정석대로 갔다. AWS 사용자 가이드에는 SAP 전용 액션 커넥터가 안내되어 있다 — Business Partner, Material Stock, Bill of Materials, Physical Inventory, Product Master. 작성 시점 문서 기준 다섯 종. 목록을 두 번 읽었다. FI/CO가 보이지 않는다. 고객이 원하는 건 CFO 보고 장표인데, 문서에 안내된 커넥터 중에는 재무 데이터로 가는 경로가 없는 것이다.

벽은 하나 더 있었다. 표준 커넥터는 Quick 클라우드가 SAP를 직접 호출하는 구조라, 데모 서버의 자체서명 TLS 인증서가 검증 단계에서 걸릴 수밖에 없다. 유효 인증서 발급, 리버스 프록시… 하루 저녁의 실험 범위를 넘어간다.

발상의 전환 — 서버를 클라우드가 아니라 내 PC에서 띄우면?

Quick 데스크탑에는 User MCP Servers 기능이 있다. 로컬 프로세스(stdio)로 MCP 서버를 등록하면, Quick의 AI가 그 서버의 도구를 쓸 수 있다. 마침 Claude Code로 개발해 온 ABAP ADT 기반 MCP 서버(abap-mcp-adt)가 있었다. 이걸 등록하면 구조가 이렇게 바뀐다.

architecture
[표준 커넥터 방식]
Quick 클라우드 ──HTTPS──▶ 온프레미스 SAP     ← TLS 검증·커넥터 종류 제약

[MCP 방식 — 이번 실증]
Quick 클라우드 ⇄ Quick 데스크탑 ⇄ MCP 서버(로컬 stdio) ──HTTPS──▶ 온프레미스 SAP
                                  └ SAP 호출이 내 PC에서 일어남
                                    · 자체서명 인증서 문제 소멸
                                    · 자격증명은 OS 키체인에만 존재
                                    · ADT 기반 → 테이블·CDS 자유 조회

등록 자체는 Command와 Arguments 두 줄이면 끝난다. 딱 하나 함정이 있었는데 — GUI 앱은 터미널과 PATH가 달라서 node라는 이름만으로는 실행 파일을 찾지 못한다. 절대경로로 바꾸자 바로 연결됐다.

Connected등록 완료 후 Quick이 인식한 도구는 174개. 테이블 조회(GetTableContents), SQL 집계(GetSqlQuery), 오브젝트 검색(SearchObject)까지 — 대시보드를 만드는 데 필요한 재료가 전부 갖춰졌다.
03여정

첫 조회에서 손익계산서까지

연결이 됐으니 이제 대화만 남았다. 데모용 테스트 서비스의 작은 조회로 연결을 검증하고, 곧바로 실제 재무 데이터로 확장했다. 아래는 그날 저녁 Quick과 실제로 주고받은 대화의 압축이다 — 모든 단계의 입력은 자연어였다.

💬 세션 로그 1 — 시작
나
“SAP 시스템과 연동해서 데이터를 읽어서 대시보드로 구현해줄 수 있나요?”
AI
“가능합니다. 방식은 ①실시간 직접 연동 ②파일 기반 ③중간 API/DB 경유로 갈립니다. 어떤 데이터를 보고 싶으신가요?”
▷ 진행된 것실시간 직접 연동 + 매출/재무(FI/CO) 선택 → 세션에 SAP 커넥터 부재 확인 → 커스텀 MCP 경로 제안.
💬 세션 로그 2 — MCP 등록
나
“혹시 제가 Claude Code로 구현한 MCP 서버 사용이 가능할까요? ADT MCP입니다.”
AI
“네, 사용자 MCP 서버 등록 방식으로 가능합니다. 실행 방식(stdio)과 진입점을 알려주시면 등록값을 정리해 드리겠습니다.”
▷ 진행된 것User MCP(stdio) 등록 → Connected · 도구 174개 인식 (GetSqlQuery · GetTableContents · SearchObject …).
💬 세션 로그 3 — 첫 조회, 연결 검증
나
“등록 완료, 연결 정상입니다.”
AI
“노출된 도구를 점검하고, 테스트 서비스의 기반 CDS 뷰를 찾아 시험 조회하겠습니다.” → “실시간 연동 성공! ZC_UZ_STUDENT 뷰에서 3건을 읽었습니다. 이 데이터로 첫 대시보드를 만들겠습니다.”
▷ SAP에서 실행된 것SearchObject('ZSTUDENT*') → 서비스 정의·CDS 구조를 스스로 파악 → 데이터 조회 → KPI·차트·테이블 대시보드 생성.
💬 세션 로그 4 — 차단, 그리고 승인
나
“FI/CO 실제 데이터로 확장해 주세요.”
AI
“표준 재무 뷰를 탐색해 ACDOCA를 조회하려 했으나, MCP 서버의 보호 목록(blocklist)에 막혔습니다. 개인정보가 연결될 수 있는 거래 데이터입니다. 개별 레코드 없이 집계 방식으로만 조회할까요?”
나
“집계 조회로 승인합니다.”
▷ SAP에서 실행된 것GetSqlQuery(집계 전용) → 32개 회사코드/연도 조합 스캔 → 데이터 최다 조합(FY2025 · 2,061라인) 선택 → FI/CO 재무 대시보드.

이 네 번째 로그가 그날의 하이라이트다. Quick은 차단을 우회하려 들지 않았다. 막힌 이유를 설명하고, 허용 가능한 대안(집계)을 제시하고, 승인을 기다렸다. AI의 자율성과 시스템의 통제가 충돌 없이 맞물린 장면이었다 — 이 통제 구조는 §05에서 자세히 다룬다.

💬 세션 로그 5 — 손익계산서, 그리고 저장
나
“P&L 손익 뷰로 재구성해 주세요.”
AI
“계정유형(GLAccountType)으로 손익 계정(P/N)만 분리하고, SAP 부호 규약(대변=수익·차변=비용)을 반영해 산출하겠습니다.” → “총수익 2,960,512 · 총비용 2,434,439 · 순손익 +526,073 (순이익률 17.8%)입니다.”
나
“지금까지 만든 대시보드 모두 HTML 파일로 저장해 주세요.”
▷ SAP에서 실행된 것손익 집계 쿼리 → P&L 대시보드 → 독립 실행형 HTML 4종 + 대화 기록 문서 저장. 세션 종료.

05통제와 검증

보안 장치, 그리고 사람

통제 — AI에게 열쇠를 주되, 방마다 잠금은 남겨둔다

“AI가 ERP 데이터를 마음대로 뒤진다”는 상상은 보안 담당자를 잠 못 들게 한다. 이번 실증에서 확인한 것은 그 반대의 그림이다. 통제 지점이 AI 바깥, 서버 쪽에 있으면 AI의 자율성과 데이터 보호가 공존한다.

  • 민감성은 조직이 정의한다 — 어떤 테이블을 어느 수준으로 보호할지는 MCP 서버의 차단 목록(blocklist)에 정책으로 등록한다. 급여 테이블을 막을지, 재무 전표를 막을지, 어디까지 허용할지 — 판단의 주체는 AI가 아니라 시스템을 운영하는 조직이다.
  • 등급에 따라 다르게 동작한다 — 완전 차단 등급의 테이블은 AI가 어떤 프롬프트를 받아도 서버가 거부한다. 승인 등급의 테이블은 사용자가 명시적으로 허가한 경우에만, 그것도 개인정보 미포함 집계로만 열린다. 승인 이력은 감사 로그로 남는다 — 이번 ACDOCA 조회가 정확히 이 경로를 밟았다.
  • 자격증명 격리 — SAP 비밀번호는 OS 키체인에만 존재. Quick에도, 채팅 기록에도, 설정 파일에도 평문이 남지 않는다.
정직한 경계선 — 다만 이것까지 통제되는 건 아니다. Quick은 SaaS이므로, MCP가 조회한 결과 데이터는 분석을 위해 AWS로 전송된다. “데이터가 외부로 나가면 안 된다”는 요구가 있다면 그 문장의 정의부터 확인해야 한다 — 퍼블릭 인터넷 미경유가 기준이라면 전용선·PrivateLink로 풀 수 있지만, 사내 경계 밖 반출 금지가 기준이라면 SaaS형 AI 도구 자체가 성립하지 않는다.

검증 — AI가 만든 손익계산서, 합계를 다시 더해보다

P&L 대시보드를 검토하다가 이상한 점을 발견했다. KPI 카드의 총수익은 2,960,512인데, 월별 차트의 수익을 전부 더하면 1,738,275 — 1,222,237이 비어 있었다. 비용도 정확히 같은 규모로 부족했다. 추적해 보니 월별 집계 쿼리에서 일부 회계기간(1·2·5월)이 통째로 빠진 것이었다. 계정 기준 집계(KPI)는 맞고, 기간 기준 집계(차트)만 누락 — 흥미롭게도 순손익은 거의 정확했기에, 합계를 직접 검산하지 않았다면 지나쳤을 것이다.

속도는 AI가 만들었지만, 신뢰는 사람의 검증이 만든다.

재무 데이터 앞에서 “그럴듯함”은 위험하다. CFO급 청중은 합계부터 다시 더한다. AI 산출물을 업무에 쓰려면 검증 루프가 프로세스의 일부여야 한다는 것 — 이번 실증이 남긴 가장 실용적인 교훈이다.

06맺음말

연결은 더 이상 장벽이 아니다

“자연어로 장표를 만든다”는 요구에는 사실 두 가지 수준이 있다. 사전에 개발된 API를 AI가 호출하는 기정의 API 호출형과, AI가 자연어를 해석해 필요한 데이터를 스스로 찾아내는 자유 탐색형. 이번 실증이 보여준 것은 후자다 — Quick은 서비스 정의를 읽고, 기반 CDS 뷰를 찾고, 집계 쿼리를 스스로 설계했다.

한 시간의 실험이 남긴 결론은 단순하다. 온프레미스 SAP와 AI 어시스턴트를 연결하는 일 자체는 이제 장벽이 아니다. 이관도, 대규모 인프라 작업도 없이 저녁 한나절에 연결이 끝났다. 남는 질문은 기술이 아니라 설계다 — 무엇을 보여줄 것인가(기능 정의), 어디까지 허용할 것인가(보안 경계), 그리고 누가 검증할 것인가(품질 루프).

이 세 가지에 답이 있는 조직이라면, SAP 데이터 위의 AI 대시보드는 프로젝트가 아니라 대화 한 세션이 될 수 있다.

예상 질문 (FAQ)

Q1. Quick의 표준 SAP 커넥터로는 왜 안 했나?

작성 시점의 AWS 공식 문서에 안내된 SAP 액션 커넥터는 Business Partner·자재 계열 5종으로, FI/CO는 목록에 없었다. 또 클라우드가 SAP를 직접 호출하는 구조라 유효한 TLS 인증서가 전제된다. 재무 대시보드 + 자체서명 인증서 환경이라는 이번 조건에서는 MCP 경로가 유일하게 저녁 안에 끝나는 길이었다.

Q2. SAP를 AWS로 이관해야 하는 것 아닌가?

아니다. AWS 공식 문서 기준으로 Quick의 SAP 연동 전제조건은 “네트워크 연결”뿐이며, 온프레미스 데이터 연결도 공식 지원 대상이다. 이번 실증도 온프레미스 서버 그대로 진행했다.

Q3. 이 MCP 서버를 그대로 도입하면 되나?

안 된다. 이번에 쓴 서버는 개인 개발 데모 구현체다. 실제 프로젝트에는 권한 모델, 감사, 운영성을 갖춘 상용 수준의 MCP 서버를 별도로 설계·구현해야 하며, 그것이 곧 개발 범위 산정의 대상이 된다.

Q4. 보안팀이 가장 먼저 물어볼 것은?

“데이터가 어디까지 가는가”다. SAP 호출은 로컬에서 일어나고 자격증명은 키체인에 격리되지만, 조회된 결과는 분석을 위해 Quick(SaaS)으로 전송된다. 도입 검토라면 이 경계를 명시하고, 필요 시 전용선·PrivateLink 구성과 데이터 최소화(집계-only) 정책을 함께 설계해야 한다.

Q5. AI가 만든 대시보드를 그대로 보고에 써도 되나?

이번 실증의 답은 “검증 후에”다. 실제로 P&L 월별 차트에서 회계기간 누락이 발견됐다. 집계 기준이 둘 이상인 산출물(계정 기준 vs 기간 기준)은 교차 검산이 필수이며, 이 검증 단계를 프로세스에 넣는 것까지가 도입이다.

이 문서의 실증은 온프레미스 S/4HANA 2023 데모 시스템(IDES 연습 데이터)에서 진행했습니다. 화면·수치는 실제 세션 산출물 원본이며, 특정 고객사의 데이터와 무관합니다. Amazon Quick, QuickSight는 Amazon Web Services의 상표입니다.
